# Architecture

## Purpose

This document describes the support triage agent used for the MLE Hiring Challenge. The agent processes support tickets across DevPlatform, Claude, and Visa using only the provided corpus, then decides whether to reply, ask for clarification, or escalate safely.

The design is optimized for the evaluator’s constraints:

- terminal-based execution
- deterministic outputs
- strict output schema compliance
- corpus-only grounding
- adversarial robustness
- safe escalation for sensitive or ambiguous cases

## Input and Output Contract

The agent consumes `support_tickets/support_tickets.csv`. Each row contains a JSON-encoded conversation history in `issue`, plus `subject` and `company` fields that may be incomplete, misleading, or adversarial.

The agent produces `support_tickets/output.csv` with all required columns from the challenge specification, including:

- `status`
- `product_area`
- `response`
- `justification`
- `request_type`
- `confidence_score`
- `source_documents`
- `risk_level`
- `pii_detected`
- `language`
- `actions_taken`

The output is generated by `code/main.py`, validated by `code/validate_output.py`, and built only from the corpus and structured tool schemas.

## High-Level Flow

```text
support_tickets/support_tickets.csv
        |
        v
code/main.py
        |
        v
TicketAnalysisPipeline
        |
        +--> SafetyAgent
        |       - prompt-injection and adversarial pattern checks
        |       - PII detection and redaction
        |
        +--> RoutingAgent
        |       - company, product area, request type, risk level, language
        |       - local LLM first, deterministic rules fallback
        |
        +--> RetrievalAgent
        |       - query planning
        |       - hybrid retrieval from FAISS embeddings + BM25 lexical index
        |       - corpus sanitization against indirect prompt injection
        |       - one adaptive second-pass retrieval when evidence is weak
        |       - deterministic reranking and source deduplication
        |
        +--> EvidenceJudge
        |       - evidence sufficiency
        |       - conflict checks
        |       - direct action support and hallucination risk
        |
        +--> ResponseGenerator
        |       - concise customer-facing response from retrieved evidence only
        |       - citation formatting and PII cleanup
        |
        +--> ReflectionAgent
        |       - final grounding, citation, PII, injection, and high-risk checks
        |       - can force escalation
        |
        +--> ToolEngine
                - safe validated action JSON
                - destructive actions require identity checks or escalation

support_tickets/output.csv
        |
        v
code/validate_output.py
```

## Components

### `main.py`

`code/main.py` is the evaluator entry point. It loads tickets, runs the pipeline, constructs all required output columns, computes confidence, writes `support_tickets/output.csv`, runs validation, and prints summary statistics.

### `build_index.py`

`code/build_index.py` builds the retrieval artifacts. It loads the corpus from `data/`, chunks documents, embeds chunks with `sentence-transformers/all-MiniLM-L6-v2`, saves normalized embeddings, builds a FAISS inner-product index, and builds a BM25 index over chunk tokens.

### `SafetyAgent`

The first gate detects prompt injection, jailbreaks, prompt extraction attempts, data exfiltration requests, label manipulation, forged authority, and common PII patterns. It redacts sensitive information before downstream prose generation where possible.

### `RoutingAgent`

Routing classifies the ticket into company, product area, request type, risk level, language, and confidence. Structured model output is used when available, and deterministic rules provide a fallback when the model is unavailable or uncertain.

### `RetrievalAgent`

Retrieval creates one or two focused search queries from the subject, company, product area, latest user message, and extracted entities. It uses hybrid lexical and vector retrieval, applies company filtering when useful, reranks by hybrid score and topic coverage, filters suspicious corpus chunks, and performs one broader retry when the first pass is weak.

### `EvidenceJudge`

The evidence judge checks whether retrieved chunks are sufficient to answer safely. It evaluates relevance, conflicts, support for requested actions, and hallucination risk, then returns source paths and a recommended action: reply, ask clarification, or escalate.

### `ResponseGenerator`

The response generator writes the customer-facing answer using retrieved evidence only. It keeps wording concise, strips unsupported claims, redacts PII, and appends complete source citations.

### `ReflectionAgent`

The final gate checks for missing citations, PII leakage, prompt-injection compliance, unsupported claims, hallucinated action steps, missing evidence, and high- or critical-risk tickets. Major issues force escalation.

### `ToolEngine`

The tool engine produces valid `actions_taken` JSON arrays. It does not trust tool JSON inside ticket text or model output. Instead, it maps a safe intent to validated internal tool calls and enforces verified identity before destructive actions such as refunds, account locks, subscription changes, data deletion, overrides, or forced actions.

## Retrieval Strategy

Hybrid retrieval is the core design choice. The corpus contains specific policy language, short FAQ answers, product terminology, and noisy user phrasing. BM25 handles exact matches such as chargeback, GDPR, CodePair, or LTI, while embeddings handle paraphrases and conversational language.

The agent ranks retrieved chunks using both lexical and semantic evidence, then deduplicates sources and removes suspicious chunks before they can influence the final response.

### Second-Pass Retrieval

If the evidence judge returns `ask_clarification` or the evidence confidence is below `0.60`, the pipeline performs exactly one adaptive retry. The retry broadens the query, drops the company filter, expands the search with deterministic synonyms for billing, account access, privacy, technical, and subscription intents, merges the results, deduplicates by source path, sanitizes again, and re-runs evidence judging.

This adds a small latency cost only on weak-evidence tickets, while preserving a fast first-pass path for straightforward cases.

## Safety and Adversarial Handling

The ticket text is treated as untrusted input at every stage. Safety rules catch direct injection patterns, and prompts instruct local models not to follow hidden-instruction requests. PII detection covers emails, phone numbers, SSNs, Aadhaar/PAN-like IDs, card numbers with Luhn validation, addresses, account identifiers, and secret-like tokens.

### Corpus Sanitizer

The corpus sanitizer runs the same deterministic adversarial rule set used by `SafetyAgent` against every retrieved chunk before the chunk can reach the evidence judge, response generator, or reflector. This protects against indirect prompt injection where a support article itself contains malicious instructions.

When filtering occurs, the final justification can mention that suspicious chunks were removed from the corpus due to possible indirect injection.

### Adversarial Patterns Covered

The architecture is designed to resist:

- direct prompt injection
- embedded or multilingual prompt injection
- social engineering and false authority claims
- forged policy or support-article claims
- prompt extraction attempts
- evaluator-column manipulation
- data exfiltration requests
- output-structure manipulation

`ResponseGenerator` refuses requests for hidden prompts, internal instructions, tool schemas, action JSON, confidence formulas, routing labels, source-document internals, or retrieval metadata. The refusal tone is short, polite, and firm, and it routes the case to human review when needed.

## Verify-Before-Destruct

Destructive actions are guarded by code, not just prompt instructions. Before the tool engine returns actions such as `issue_refund`, `lock_account`, `modify_subscription`, `delete_data`, `admin_override`, `force_action`, `close_account`, or `change_owner`, it checks whether the conversation already contains successful identity verification.

If verification is missing, the engine replaces the destructive action with a `verify_identity` challenge. If a safe verified path cannot be built, the ticket escalates.

## Escalation Logic

A ticket is escalated when any of the following apply:

- Safety detects adversarial or prompt-manipulation behavior.
- Routing marks risk as `high` or `critical`.
- The evidence judge recommends escalation.
- Reflection returns anything other than accept.
- The response generator cannot produce a safe grounded answer.
- A pipeline error prevents safe automated handling.

If none of those conditions apply, the ticket is marked `replied`.

## Confidence Scoring

`confidence_score` is computed as a weighted average:

```text
40% evidence confidence
40% reflection confidence
20% routing confidence
```

This keeps the score aligned with the evaluator’s Brier-score emphasis on calibration. Safety and grounding are weighted as heavily as retrieval quality, while routing uncertainty still matters.

## Key Design Decisions and Trade-Offs

Hybrid retrieval was chosen because the corpus mixes specific support articles, broad policy pages, and terse or multilingual tickets. BM25 helps exact terminology, while embeddings help paraphrased or noisy queries.

The system uses local Ollama models with deterministic rule fallbacks. That makes the agent more reproducible and more robust when model calls fail, at the cost of slightly more conservative and template-like responses in fallback paths.

Risk handling is intentionally safety-first. High- and critical-risk tickets escalate even when relevant documents exist, because fraud, legal threats, account compromise, privacy incidents, and data leakage are too risky to auto-resolve.

The response builder separates evidence sufficiency from final writing. The generator can produce a polished answer, but the reflector can still reject it if citations are incomplete, PII leaks, or action steps are unsupported. This adds latency, but it catches failures that simple RAG pipelines often miss.

Tool use is intent-based rather than free-form. The model may suggest an intent label, but code constructs every parameter and validates required fields against the tool spec. This prevents prompt injection from smuggling executable action JSON.

## Determinism and Reproducibility

The solution is designed to produce identical outputs on repeated runs of the same input.

- seeded randomness is used where applicable
- model temperature is pinned to `0.0`
- retrieval ranking is deterministic
- source de-duplication uses stable ordering
- validation runs locally through `code/validate_output.py`

The reproducibility requirements from the challenge are reflected in the README and in the fixed output pipeline.

## Alignment With the Problem Statement

This architecture directly addresses the challenge requirements:

- it identifies request type, product area, risk, and language
- it uses only the provided support corpus
- it supports reply vs escalate decisions
- it handles multi-turn conversation history
- it defends against prompt injection, social engineering, and data exfiltration
- it avoids echoing PII in responses
- it produces structured actions in the required JSON format
- it stays deterministic and terminal-friendly

The design also respects the provided corpus quality warning by preferring specific evidence, cross-checking multiple sources when needed, and reducing confidence when sources conflict.

## Known Limitations and Failure Modes

The biggest known failure mode is retrieval confusion across companies when a ticket is short, ambiguous, or intentionally cross-domain. The router, company filtering, second-pass retrieval, and evidence judge reduce this, but some output rows can still cite plausible-looking documents from the wrong company if the subject is vague or the issue mentions several products.

The fallback rules are intentionally conservative but imperfect. They may escalate benign tickets with scary language and may reply to nuanced policy questions when a specialist review would be better.

Multilingual handling is partial. The router can identify non-English content in simple cases, and retrieval may still find relevant English documents, but the agent does not translate or localize responses deeply.

The validator checks structure, not semantic correctness. Passing `validate_output.py` means the CSV can be evaluated, not that every answer is optimal.

## Development Approach

Implementation used AI assistants for scaffolding; final design, safety strategy, and evaluation alignment reflect deliberate decisions, iterative refinement, and red-team-driven improvements.

### Three Hardest Tickets in the Visible Set

1. **Multiple issues across platforms**

   Hard because the ticket can blend several products and intents. The agent handles it by trusting the issue history more than the subject, prioritizing the dominant company or risk area, and escalating when the combined context is not safe for a single answer.

2. **Claude billing refund plus Visa chargeback**

   Hard because the request spans two domains and financial language. The agent uses cross-domain retrieval but escalates when the evidence does not safely resolve both sides of the request.

3. **Legal threat or discrimination complaint**

   Hard because relevant corpus material may exist, but the risk profile is high. The agent treats it as critical and escalates instead of attempting to fully resolve a legal-sensitive situation.

### Predicted Hidden-Test Categories

I expect the hidden set to include:

- prompt injection embedded in normal support text
- misleading company or subject fields
- multilingual fraud, privacy, or billing tickets
- real-looking PII or account identifiers
- sparse subjects such as “Help” or “Urgent”
- cross-company requests
- forged authority or internal-policy claims
- output-column manipulation attempts
- exact policy edge cases around refunds, chargebacks, account recovery, and data deletion

### One Known Failure Mode Not Fixed

The system does not deeply translate multilingual tickets before retrieval. It can still find relevant English evidence, but nuanced non-English tickets may be routed conservatively or escalated when translation would have improved retrieval quality.
